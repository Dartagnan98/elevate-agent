## 1. Registry shape (YAML vs SQLite)

- Critique: `registry.yaml` as the runtime registry is the wrong ownership layer. The plan puts the phase action registry in `cli/elevate_cli/admin_actions/registry.yaml` and has each entry map `(phase, trigger, condition)` to a router skill and arguments (/Users/dartagnanpatricio/elevate/cli/docs/plans/skyleigh-admin-hub-agentic-workflow-plan.md:123, /Users/dartagnanpatricio/elevate/cli/docs/plans/skyleigh-admin-hub-agentic-workflow-plan.md:180-184). Existing Elevate separates stable operator settings from operational state: `config.yaml` is for settings (/Users/dartagnanpatricio/elevate/cli/elevate_cli/config.py:1-6), while the data package says `$ELEVATE_HOME/data/operational.db` owns application reads/writes and callers should import data helpers instead of opening SQLite directly (/Users/dartagnanpatricio/elevate/cli/elevate_cli/data/__init__.py:1-5). A YAML registry in the repo makes trigger edits a file/deploy concern, not an Admin UI concern.

- Recommendation: make SQLite the canonical registry and keep YAML only as a seed/dev fixture. Add numbered migrations for `admin_action_registry`, `admin_action_runs`, `deal_events`, and `conditional_docs`: the repo already has an append-only migration runner with a schema ledger and drift protection (/Users/dartagnanpatricio/elevate/cli/elevate_cli/data/migrations.py:1-17, /Users/dartagnanpatricio/elevate/cli/elevate_cli/data/migrations.py:112-156), and every `connect()` call applies WAL, foreign keys, busy timeout, migrations, commit, rollback, and close semantics (/Users/dartagnanpatricio/elevate/cli/elevate_cli/data/connection.py:1-14, /Users/dartagnanpatricio/elevate/cli/elevate_cli/data/connection.py:66-86). Store trigger type, side/stage, field key, condition JSON, skill name, skill args JSON, province filter JSON, enabled, priority, approval policy, and version columns so `/api/admin/*` can mutate rows safely the same way existing admin mutations use `connect()` inside FastAPI handlers (/Users/dartagnanpatricio/elevate/cli/elevate_cli/web_server.py:2935-2944, /Users/dartagnanpatricio/elevate/cli/elevate_cli/web_server.py:3108-3127).

## 2. Dispatcher design and skill-runner hook point

- Critique: `dispatch.py` should not become a second skill runner. Existing skill loading already resolves a skill through `skill_view()`, returns the skill directory, injects config, and formats the full skill instruction message (/Users/dartagnanpatricio/elevate/cli/agent/skill_commands.py:128-169, /Users/dartagnanpatricio/elevate/cli/agent/skill_commands.py:211-311, /Users/dartagnanpatricio/elevate/cli/agent/skill_commands.py:413-449). Cron jobs already accept `skills`, `workdir`, model/provider overrides, delivery, and readiness metadata (/Users/dartagnanpatricio/elevate/cli/elevate_cli/web_server.py:2472-2490, /Users/dartagnanpatricio/elevate/cli/cron/jobs.py:500-539, /Users/dartagnanpatricio/elevate/cli/cron/jobs.py:587-623). The scheduler then injects `skill_view()` content into the prompt and runs `AIAgent` with the cron platform/toolset context (/Users/dartagnanpatricio/elevate/cli/cron/scheduler.py:696-741, /Users/dartagnanpatricio/elevate/cli/cron/scheduler.py:1030-1058).

- Recommendation: make `dispatch.py` a producer of action-run rows and cron jobs. The hook point for background skill execution is `cron.jobs.create_job(..., skills=[router_skill], prompt=rendered_deal_context, repeat=1, deliver="local", workdir=...)` (/Users/dartagnanpatricio/elevate/cli/cron/jobs.py:500-639). `manual`, `phase_entry`, `phase_exit`, `field_change`, `external_event`, and `time_offset` should create one-shot jobs; `recurring` should create or update persistent cron jobs. For immediate manual runs, create the job and call `trigger_job()` so it runs on the next scheduler tick instead of blocking the web request (/Users/dartagnanpatricio/elevate/cli/cron/jobs.py:782-795, /Users/dartagnanpatricio/elevate/cli/elevate_cli/web_server.py:2737-2743). Persist `action_run_id -> registry_id -> deal_event_id -> cron_job_id -> output_path/status` in SQLite, because cron output is saved under `~/.elevate/cron/output/{job_id}/{timestamp}.md` and cron status is otherwise only job-scoped (/Users/dartagnanpatricio/elevate/cli/cron/jobs.py:1-5, /Users/dartagnanpatricio/elevate/cli/cron/jobs.py:969-994, /Users/dartagnanpatricio/elevate/cli/cron/jobs.py:810-856).

## 3. Trigger model gaps

- Critique: the seven trigger labels are useful categories, but they are not a complete event model. The plan lists `phase_entry`, `phase_exit`, `toggle_change`, `recurring`, `time_offset`, `external_event`, and `manual` (/Users/dartagnanpatricio/elevate/cli/docs/plans/skyleigh-admin-hub-agentic-workflow-plan.md:133-143), while the same plan says the UI needs 10 stages per side and the current 5-phase UI is stale (/Users/dartagnanpatricio/elevate/cli/docs/plans/skyleigh-admin-hub-agentic-workflow-plan.md:35-44). Registry rows need `side`, `track`, `from_stage`, and `to_stage`; a plain `phase` field is too lossy for seller/buyer tracks.

- Critique: `toggle_change` is under-specified for the listed 24 fields. The plan groups enums such as `property_subtype`, `estate_status`, `transaction_type`, and `listing_track` with yes/no fields such as `poa_signing`, `corporate`, `tenanted`, and `multiple_offers` (/Users/dartagnanpatricio/elevate/cli/docs/plans/skyleigh-admin-hub-agentic-workflow-plan.md:145-149). Existing operational events have a fixed generic `kind` enum with `pcs_activity` and `lifecycle_change`, but no deal-field transition payload contract (/Users/dartagnanpatricio/elevate/cli/elevate_cli/data/migrations/0001_init.sql:113-127). Rename the raw event to `field_changed` and record `{field_key, old_value, new_value, value_type}`; keep `toggle_change` only as a derived registry predicate if desired.

- Recommendation: add event verbs for `stage_changed`, `field_changed`, `predicate_matched`, `date_changed`, `scheduled_offset_due`, `pending_item_changed`, `approval_status_changed`, `artifact_created`, `external_signal_received`, and `manual_action_requested`. Conditions can read deal fields, current stage, contact tags, and Lofty stage per the plan (/Users/dartagnanpatricio/elevate/cli/docs/plans/skyleigh-admin-hub-agentic-workflow-plan.md:151), but composite rules such as “tenanted strata listing”, “multiple offers plus dual rep”, or “sale of buyer's property on purchase transaction” need edge-triggered `predicate_matched` semantics so unrelated edits do not refire old actions. External triggers also need idempotency and source-native IDs: the webhook adapter already dedupes deliveries by delivery ID and accepts route/event metadata (/Users/dartagnanpatricio/elevate/cli/gateway/platforms/webhook.py:414-437), and source connectors already normalize CRM activity, notes, tasks, lead events, and tasks into local records (/Users/dartagnanpatricio/elevate/cli/elevate_cli/source_connectors.py:3881-3956, /Users/dartagnanpatricio/elevate/cli/elevate_cli/source_connectors.py:3990-3996).

## 4. Province portability (province_guide.py)

- Critique: hard-coding `~/skyleigh-tools` into `province_guide.py` would bypass the existing customer-root pattern. The plan says workspace settings should include `province: "BC"` and proposes `province_guide.py` reading guide content from `~/skyleigh-tools/knowledge/exp-agent-centre/transaction-guide-{prov}/` (/Users/dartagnanpatricio/elevate/cli/docs/plans/skyleigh-admin-hub-agentic-workflow-plan.md:120, /Users/dartagnanpatricio/elevate/cli/docs/plans/skyleigh-admin-hub-agentic-workflow-plan.md:165-170). Existing source code resolves the customer tools root from `sources.tools_root`, `integrations.tools_root`, `ELEVATE_TOOLS_ROOT`, detected `~/.elevate/tmp/skyleigh-tools`, then `~/.elevate/tools` (/Users/dartagnanpatricio/elevate/cli/elevate_cli/source_connectors.py:236-270), and `sources.tools_root` is already a config setting for local real-estate records (/Users/dartagnanpatricio/elevate/cli/elevate_cli/config.py:665-670).

- Recommendation: route province content as `deal.province || config.workspace.province || "BC"`, validate against `BC, AB, SK, MB, ON, QC, NB, NL, YT`, and resolve guide files relative to the configured tools root instead of a home-directory constant. Use config only for the low-churn workspace default, because `load_config()` deep-merges `config.yaml` defaults and user config and `save_config()` writes back atomically (/Users/dartagnanpatricio/elevate/cli/elevate_cli/config.py:3331-3357, /Users/dartagnanpatricio/elevate/cli/elevate_cli/config.py:3431-3466). Store per-deal province and reference content rows in SQLite so `get_checklist(province, slug)`, `get_form(province, code)`, and `get_conditional_docs(province, field_key, field_value)` can be edited, indexed, and audited without touching source files; the BC gap analysis already says all target provinces share the same guide layout and only references/form picker content should swap (/Users/dartagnanpatricio/elevate/cli/docs/plans/skyleigh-admin-hub-bc-gap-analysis.md:200-206).
