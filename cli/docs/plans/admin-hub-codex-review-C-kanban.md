## 1. Component changes

- The widening is not just a CSS change. The component currently models the board as `AdminLane = "seller" | "buyer" | "other"` and each card only stores `columnId`, so deal side and workflow position are inferred from a column lookup instead of being first-class card state (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:4784`, `/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:4796-4808`). For 10 stages x 2 lanes, change the component state shape to make `side: "listing" | "buyer"` and `stage: 0 | 1 | ... | 9` explicit on each deal/card.

- `ADMIN_COLUMNS` currently defines five seller columns, five buyer columns, and one `other-misc` column (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:4810-4824`). Replace that with either 20 side-specific stage definitions or a 10-stage array with side-specific labels/checklists. Do not keep `other` as a swimlane if the target is strictly Listing/Buyer; the current board still renders an `Other Items` swimlane (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5543-5551`).

- The current `phaseRef` field is visible board metadata, not an internal-only comment: it is part of `AdminColumn` and renders in both column headers and side-panel phase sections (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:4786-4792`, `/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5125-5127`, `/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5281-5283`). Rename this around stages, for example `stageLabel`, `stageNumber`, and optional `sourcePhaseRef`, so the UI stops presenting broad 5-phase buckets as the primary model.

- `ADMIN_PHASE_CHECKLISTS` is keyed by the current broad column ids (`s-cma`, `s-prelist`, `b-walk`, etc.), and progress reads only `card.columnId` (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:4827-4917`, `/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:4935-4949`). A 10-stage board needs checklist keys scoped by side and stage, not by the old broad column ids; otherwise Stage 1/2 and Stage 4/5 stay collapsed into the old five-column buckets.

- The helper logic currently derives lane from `ADMIN_COLUMNS` and advances to the next column inside that lane (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:4919-4933`). With first-class `side` and `stage`, `adminNextColumn` should become a stage increment helper, stop at Stage 9, and stop special-casing `other`.

- Prop shape should change with the state shape. `AdminKanbanColumn` currently receives a full `AdminColumn` object plus already-filtered `cards`, while `AdminKanbanSwimlane` re-filters columns by `lane` internally (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5107-5115`, `/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5144-5162`). For the widened board, pass explicit `{ side, stage, cards, onCardClick, onMoveCard }` props so column rendering does not depend on hidden `columnId` parsing.

- Phase column rendering will become too wide if the current minimums are copied unchanged. Each column is `min-w-[14rem]`, and each swimlane grid repeats `laneColumns.length` with `minmax(14rem, 1fr)` (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5117`, `/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5174-5176`). For 10 columns per lane, keep horizontal scrolling but use compact stage headers and tighter fixed/min widths; otherwise each lane becomes roughly twice as wide as the current already-scrollable lane.

- `AdminCardDetailPanel` currently expands all columns for the card's lane by computing `laneColumns` and mapping every column into `AdminCardPhaseSection` (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5351-5358`, `/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5464-5480`). After widening, this panel should render the 10 stages for the selected card's side, with current/past logic based on numeric `stage` rather than array position in a mixed column catalog.

- Keep TOP 25 as a cross-lane filter, not as a third workflow lane. The existing strip already treats pinned cards as still living in their real phase column (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5188-5202`), which fits the 2-lane model better than the current `other` swimlane.

## 2. Seed-data removal

- The component is still seeded client-side: the comment calls `ADMIN_CARDS_SEED` "visual scaffolding", the seed array is static, and `AdminKanbanBoard` initializes `useState<AdminCard[]>(ADMIN_CARDS_SEED)` (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:4951-5026`, `/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5490-5492`). The plan already calls for replacing that seed plus `setCards` with `useAdminCards()` and REST-backed card moves (`/Users/dartagnanpatricio/elevate/cli/docs/plans/admin-hub-agentic-workflow-plan.md:160-164`).

- Current checklist toggles mutate only local React state and emit no deal event (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5496-5509`). Current moves also mutate only `columnId` locally (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5511-5520`). That matches the plan's warning that the kanban has no persistence and does not know the server exists (`/Users/dartagnanpatricio/elevate/cli/docs/plans/admin-hub-agentic-workflow-plan.md:41-44`).

- Component-side backend pattern: add a local `useAdminCards`/`useAdminDeals` hook that fetches on mount, tracks loading/error, and refreshes after mutations. The same file already uses this pattern for hub data with `useCallback`, `Promise.allSettled`, and a guarded `useEffect` mount load (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:127-186`). The component already imports the central `api` object, so the kanban should use that path rather than raw ad hoc fetch calls (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:77-78`).

- Move pattern: compute the target `{ side, stage }`, optimistically update the card, call the planned move endpoint, and roll back or refresh on failure. The plan names `/api/admin/deals`, `/api/admin/deals/:id/move`, and `/api/admin/deals/:id/toggle`, and says every stage change, toggle change, or pending-item completion should emit `deal_events` (`/Users/dartagnanpatricio/elevate/cli/docs/plans/admin-hub-agentic-workflow-plan.md:121-127`). This applies whether movement remains the current side-panel "Move card" button or becomes drag-and-drop; today the only move UI is the side-panel button plus `handleMoveToNext` (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5446-5459`, `/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5511-5520`).

- Toggle pattern: optimistic flip in component state, POST the changed field or checklist item to the planned toggle endpoint, then reconcile with the server response. Do not leave `completedByPhase` as the only persisted shape; it is currently a nested client-only overlay keyed by phase/column id (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:4806-4807`, `/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5496-5506`).

## 4. Toggle UI

- Recommendation: put toggles/enums in the existing side panel, not inline on cards and not in a separate modal. The plan says the per-deal toggle set drives Conditional Docs and dispatcher behavior (`/Users/dartagnanpatricio/elevate/cli/docs/plans/admin-hub-agentic-workflow-plan.md:145-151`), while the card surface is already a compact summary with initials, client, property, next deadline, and progress (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5037-5104`). Inline controls would overload the card and make stage scanning worse.

- The side panel is already the per-card editing surface: it opens as an aside, owns the selected card context, and renders editable checklist rows in a scrollable body (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5385-5394`, `/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5464-5480`, `/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5310-5313`). Add a "Conditions" or "Deal facts" section there, grouped into enums and yes/no toggles, so flips can share the same optimistic-toggle/event path as checklist items.

- A modal is the weaker fit because these fields need to be reviewed alongside the current stage, pending items, and next move prompt. The current panel already displays the current lane/column and move prompt in context (`/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5395-5403`, `/Users/dartagnanpatricio/elevate/cli/web/src/pages/RealEstateHubPages.tsx:5446-5461`).

- Before building the toggle UI, fix the plan schema count. The plan says "24 fields" but the visible list contains 7 enum keys and 12 yes/no keys, so only 19 keys are actually specified in the doc (`/Users/dartagnanpatricio/elevate/cli/docs/plans/admin-hub-agentic-workflow-plan.md:145-148`). The missing five keys need names before the UI can bind stable fields or emit reliable toggle-change events.
