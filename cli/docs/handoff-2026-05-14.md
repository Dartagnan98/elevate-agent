## Open issues the user surfaced this session (not all UI)

1. **Backend agent loop hangs.** `prompt.submit` reaches the gateway but no streaming events come back. `session.interrupt` does not unstick it. This is NOT a frontend bug — the agent runtime in `cli/tui_gateway/` or `cli/agent/` is blocking before emitting events. The user was seeing gpt-5.5 sessions spin forever with no reasoning / tool deltas. Diagnose by tailing the gateway log on the next hang and looking for the request arriving with no subsequent emit.
2. **Steer does not persist across refresh.** Frontend now mirrors the steered text into the transcript (`8cb...`), but the backend does not log the steered prompt into the session log. Real fix lives in the Python steer handler (search for `steer` in `cli/tui_gateway/` and `cli/elevate_cli/commands.py:97`).
3. **Subagent activity invisible in chat.** When the assistant calls the Task tool to spawn a subagent, the parent's `tool.start` lands but child agent events never reach the wire. This is a backend instrumentation gap — fix would emit nested `tool.progress` or sibling stream channel events.
4. **Sidebar `is_active` lags real busy state.** App.tsx polls `getSessions` every 12s; the dot for the active chat does not flip to the `Loader2` spinner until the next poll. Possible fix: shared `liveSessionIds` set pushed up from `ChatPage` via a context or callback when streaming starts/stops, OR have ChatPage proactively trigger a sidebar reload on busy state change.

## Dead / partially-wired UI from this session

These UI changes ship in the bundle but are missing backend wiring or have UX gaps. Wire them up in the new chat before treating the chat experience as production-ready.

### 1. Steer transcript append (commit `8cb...`)
- **Frontend**: ChatPage.tsx now calls `appendMessage("user", item.text)` on `session.steer` success.
- **Missing**: backend does NOT persist the steered text into the session transcript file. After a refresh, the message vanishes.
- **Wire-up needed**: in the Python steer handler (search for `session.steer` in `cli/tui_gateway/`), log the steered text as a user-role message into the session's append-only transcript log, the same way `prompt.submit` does.

### 2. Queue persistence (`elevate.chat.queueCache.v1` in localStorage)
- **Frontend**: `writeQueue` / `restoreQueue` in ChatPage.tsx mirror `queuedInputs` to localStorage keyed by sessionId, restored in the resume effect.
- **Missing**: if the user closes the tab with items still queued (not yet sent), nothing on the backend knows about them. Reopening surfaces them in the UI but they will not auto-dispatch unless the user resubmits them.
- **Wire-up needed**: either (a) backend accepts a `queue` array in `session.resume` and replays them, or (b) ChatPage auto-dispatches restored queue items via `prompt.submit` once the session reopens. (b) is simpler and frontend-only — there is already a `queueDispatchRef` that could be repurposed.

### 3. Subagent activity in chat
- **Frontend**: `gw.on("thinking.delta" / "reasoning.delta" / "tool.start" / "tool.progress" / "tool.complete")` handlers fan-out into the transcript via `addActivityTrace` / `ToolCall` rows.
- **Missing**: backend Task tool calls emit `tool.start` for the PARENT call but no events for the child subagent's reasoning, tool calls, or final response. The user sees an opaque "running Task" with no idea what the subagent is doing.
- **Wire-up needed**: in the Python Task-tool dispatcher, forward the child agent's stream events as nested `tool.progress` payloads (or as a sibling channel like `subagent.delta` keyed by parent tool_id). Frontend `ToolCall` already supports `preview` updates from `tool.progress`.

### 4. Session unread / needs-attention flag (SessionStatusDot warning state)
- **Frontend**: SessionStatusDot in App.tsx renders yellow when `unreadIds` includes the session.
- **Missing**: `unreadIds` only flips when the user manually right-clicks "Mark as unread". There is no automatic trigger when a session legitimately needs attention (e.g. tool failure, waiting-for-approval, agent error).
- **Wire-up needed**: backend should emit a `session.needs_attention` event (or set `session.unread` on the SessionInfo payload) when a session is in a state that requires the user. Frontend would merge backend `unread` flag with the local `unreadIds` set.

### 5. Sidebar running indicator (`is_active` Loader2)
- **Frontend**: SessionRow shows `Loader2` when `session.is_active === true`, else SessionStatusDot.
- **Missing**: `is_active` only refreshes every 12s via `getSessions` polling. When the user sends a prompt, the sidebar dot stays "done" green until the next poll fires.
- **Wire-up needed**: cheapest path is frontend-only — add a `liveSessionIds` context bumped by ChatPage when `busy` flips true/false, and have SessionRow OR-combine it with backend `is_active`. Heavier path: gateway pushes a `sessions.snapshot` event on every state change and App.tsx merges incrementally.

### 6. Stale status pill at chat bottom (`statusText`)
- **Frontend**: WorkingStatusPill shows `statusText` ("Steer delivered", "Ready", "Working...", etc.).
- **Missing**: terminal states like "Steer delivered" / "Steer rejected" / "Voice captured" persist indefinitely until the next state change. If the user does nothing afterwards, the stale text lingers and looks like the system is still doing something.
- **Wire-up needed**: terminal states should auto-clear to "Ready" after ~2.5s. Easy fix: when setStatusText is called with a known terminal label, schedule a setTimeout to reset to "Ready" if statusText hasn't changed in the meantime.

### 7. Steer rejected UX
- **Frontend**: on `status === "rejected"` response, the queued item flips to `status: "error"` and statusText shows "Steer rejected".
- **Missing**: no explanation of why it was rejected (turn already complete? backend offline? capability gap?), no retry affordance, no way to recover the steered text.
- **Wire-up needed**: backend `session.steer` should return a `reason` string on rejection. Frontend should surface that reason inline on the queued item and offer "Send as new prompt" as a one-click recovery.
